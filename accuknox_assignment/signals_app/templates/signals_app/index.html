1. Synchronicity Proof

Q1

By default, Django signals are executed synchronously. When a signal is sent, Django waits for all connected receivers to finish executing before continuing the caller thread.

Proof Logic (views.py / signals.py) 
# signals.py
@receiver(post_save, sender=TestModel)
def signal_handler(sender, instance, **kwargs):
    if instance.name == 'sleep_test':
        time.sleep(5)  # Simulate delay

# views.py (Test execution)
start = time.time()
TestModel.objects.create(name="sleep_test")
end = time.time()
self.assertGreaterEqual(end - start, 5.0)

2. Threading Proof

Q2

Django signals run in the same thread as the caller by default. No new threads are spawned when dispatching signals unless explicitly initiated by the developer.

Proof Logic (signals.py / views.py) 
# Inside views.py (Caller)
caller_tid = threading.current_thread().ident

# Inside signals.py (Receiver)
signal_tid = threading.current_thread().ident

# Verification
self.assertEqual(caller_tid, signal_tid)

3. Transaction Scope Proof

Q3

Django signals execute in the same database transaction as the caller. If the outer transaction is rolled back, all DB operations performed inside the signal receiver are also rolled back.

Proof Logic (atomic scope verification) 
# views.py
with transaction.atomic():
    Order.objects.create(name="Laptop")
    # order_save signal creates Log record...
    raise Exception("Forced Rollback")

# Post-exception verification:
self.assertEqual(Order.objects.count(), 0)
self.assertEqual(Log.objects.count(), 0)

4. Python Custom Class

Class

An iterable Python class that takes length and width, and yields length followed by width in a dictionary format.

Implementation (rectangle.py) 
class Rectangle:
    def __init__(self, length: int, width: int):
        self.length = length
        self.width = width

    def __iter__(self):
        yield {"length": self.length}
        yield {"width": self.width}
